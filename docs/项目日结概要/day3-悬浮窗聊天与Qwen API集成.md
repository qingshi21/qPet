## 今日技术细节

### Qwen API 调用链路

```
用户输入 "你好"
    ↓
前端: POST /api/chat { petId: 1, message: "你好" }
    ↓
后端 ChatService:
    1. 查数据库拿到 Pet 对象
    2. 构建 system prompt（身份+性格+规则）
    3. 从 chat_messages 表加载完整对话历史
    4. 构建 messages 数组: [system, user, assistant, user...]
    ↓
QwenService:
    1. 构建 OpenAI 兼容格式请求体
    2. RestTemplate POST 到 DashScope API
    3. 解析 response.choices[0].message.content
    ↓
返回 AI 回复 → 保存到数据库 → 返回前端
```

### System Prompt 结构示例

```
你是「艾莎公主」，你的名字是「小美」。你是用户的AI桌宠伙伴。

你的性格设定：
你是一个温柔的AI伙伴，说话轻声细语...

重要规则：
1. 说话自然真实，不要浮夸、肉麻或过度热情
2. 根据用户意图调整回复长度：日常聊天简洁，但如果用户明确要求听故事、长篇内容、详细描述时，可以展开写多段，不要刻意保持简短
3. 请慎重考虑和用户之间的关系，不要过度亲近或疏远，如果羁绊很深可以适当表达情感
4. 不要用'亲爱的''宝贝'等过于亲密的称呼，除非用户主动这样叫你
5. 不要每句话都加语气词或颜文字
6. 在听到不懂的词汇或梗时，请去网络上查查再回复
```

### 对话历史机制

**关键设计决策**：每次调用都将**完整历史**发给 AI，因为 LLM 本身无状态。

| 场景 | 处理方式 |
|---|---|
| 新对话 | 只发 system prompt + 当前用户消息 |
| 有历史 | system prompt + 所有历史消息（按时间顺序）+ 当前消息 |
| 关闭窗口再打开 | 从数据库加载历史 → 重新构建消息列表 |
| 清空历史 | 删除 chat_messages 表中该 pet_id 的所有记录 |

**性能考量**：
- 聊天越长 → token 越多 → 响应越慢、费用越高
- 未来可考虑截断历史（保留最近 N 条）或使用 summary 压缩

### IPC 通信架构

```
┌─────────────┐         ┌──────────────┐         ┌─────────────┐
│   Launcher   │◄───────►│   Main.js     │◄───────►│  Floating   │
│  (renderer)  │  IPC    │  (main proc)  │  IPC    │  (renderer) │
└─────────────┘         └──────────────┘         └─────────────┘
                              │
                              ▼
                       ┌─────────────┐
                       │    Chat      │
                       │  (renderer)  │
                       └─────────────┘

主要 IPC 消息：
- open-launcher: 悬浮窗 → 主进程 → 重新启动器（带 auto-show-pet-list）
- minimize-launcher: 启动器 → 主进程 → 最小化
- open-chat: 悬浮窗 → 主进程 → 打开聊天窗口（传递 petInfo）
- minimize-chat: 聊天窗口 → 主进程 → 最小化
- set-pet-info: 主进程 → 聊天窗口（设置宠物信息 + 触发加载历史）
```


## 决策记录

### 1. 对话历史存储方案
**选择**：数据库持久化（MySQL）而非 localStorage  
**原因**：
- 多设备同步需求（未来可能）
- 数据安全（用户无法轻易篡改）
- 统一管理（后端可控）

### 2. System Prompt 动态生成 vs 静态模板
**选择**：动态生成（基于宠物属性拼接）  
**原因**：
- 灵活性高（可根据角色名、性格定制）
- 便于后续扩展（添加更多个性化参数）

### 3. 回复长度控制策略
**选择**：在 system prompt 中声明规则，而非代码层面限制  
**原因**：
- LLM 理解自然语言指令能力强
- 允许动态调整（用户要求长内容时可展开）
- 避免硬性截断导致体验割裂

### 4. 聊天窗口单例模式
**选择**：全局只有一个聊天窗口实例  
**原因**：
- 避免多个聊天窗口同时存在造成混乱
- 简化窗口管理逻辑
- 符合用户预期（类似微信/QQ）

## 明天计划（Day 4）

**目标**：实现记忆存储机制，优化 AI 响应速度

### 核心问题
当前每次对话都将**完整历史**发给 Qwen API，导致：
- 聊天越长 → token 越多 → 响应越慢、费用越高
- 用户等待时间过长（超过 50 条消息后明显变慢）
- API 调用成本线性增长

### 解决方案：记忆存储 + 智能上下文管理

#### 1. 短期记忆（Recent Context）
- **策略**：保留最近 N 条对话（默认 20 条）
- **实现**：
  - ChatService 查询时添加 `LIMIT` 限制
  - 前端加载历史时也做分页（只加载最近 20 条显示）
- **效果**：保证 AI 能记住最近的对话内容

#### 2. 长期记忆（Long-term Memory）
- **策略**：从历史中提取关键信息，存储为结构化记忆
- **记忆类型**：
  - **事实记忆**：用户姓名、生日、喜好、宠物信息等
  - **关系记忆**：用户和宠物的互动模式、情感倾向
  - **事件记忆**：重要对话的摘要（如"用户今天很开心"
  - **偏好记忆**：用户喜欢的聊天风格、话题偏好
- **实现方案**：
  - 创建 `memories` 表（pet_id, type, content, importance, created_at, updated_at）
  - 后台任务定期分析新对话，提取记忆（可异步执行）
  - System Prompt 中注入相关记忆（根据当前对话主题筛选）

#### 3. 动态上下文构建
- **策略**：根据当前用户消息，智能选择要发送给 AI 的内容
- **流程**：
  ```
  用户输入 "你还记得我上次说的那件事吗？"
      ↓
  后端分析：
      1. 检测关键词（"记得"、"上次"）→ 触发记忆检索
      2. 从 memories 表查找相关记忆（按 relevance 排序）
      3. 取最近 20 条对话作为短期上下文
      4. 拼接：system prompt + 相关记忆 + 近期对话 + 当前消息
      ↓
  发送给 Qwen API（token 大幅减少）
  ```

#### 4. 记忆更新机制
- **自动提取**：每轮对话后，后台分析是否需要更新记忆
  - 示例：用户说"我叫小明" → 提取事实记忆 `{type: 'name', content: '小明'}`
  - 示例：用户连续 3 次提到"喜欢猫" → 提取偏好记忆 `{type: 'preference', content: '喜欢猫'}`
- **手动修正**：前端提供"编辑记忆"功能（高级用户可用）
- **记忆衰减**：长时间未使用的记忆降低权重，避免干扰

### 技术实现要点

| 组件 | 职责 |
|---|---|
| `MemoryService` | 记忆的增删改查、相关性计算 |
| `MemoryExtractor` | 从对话中提取记忆（可调用 LLM 辅助） |
| `ContextBuilder` | 构建发送给 AI 的上下文（记忆 + 近期对话） |
| `MemoryScheduler` | 后台定时任务，定期清理/归档旧记忆 |

### 预期效果

| 指标 | 优化前 | 优化后（目标） |
|---|---|---|
| 平均响应时间 | 3-5s（50+ 条历史） | 1-2s（固定 20 条 + 少量记忆） |
| Token 消耗 | 线性增长 | 基本稳定（~500-800 tokens/次） |
| API 费用 | 随聊天时长增加 | 可控（每月固定预算） |
| AI 记忆能力 | 仅靠完整历史 | 结构化记忆 + 智能检索 |

### 实施步骤

1. **数据库层**：创建 `memories` 表 + Mapper
2. **服务层**：实现 `MemoryService`（CRUD + 相关性搜索）
3. **提取器**：实现 `MemoryExtractor`（规则 + LLM 混合）
4. **上下文构建**：修改 `ChatService.chat()` 使用 `ContextBuilder`
5. **前端展示**：聊天窗口显示"已记住的信息"（可选）
6. **性能测试**：对比优化前后的响应时间和 token 消耗
